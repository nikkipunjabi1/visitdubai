# Content Model — Visit Dubai

_Draft. Reconciled with the base-type names the forked demo uses (Optimizely SaaS: `_Page`,
`_Component`, `_Experience`, `_Section`, `_metadata`, etc.)._

## Modeling philosophy for Visual Builder
- **Pages** — routable content with a URL + SEO metadata.
- **Experiences** — Visual-Builder-composable pages; authors drag **Sections** + **Components**
  onto a canvas. Rendered by recursively walking the composition node tree.
- **Sections** — layout containers (grid: rows/columns) inside an experience.
- **Components (Elements)** — reusable building blocks, each mapped 1:1 to a React component in
  the factory registry.
- **Data content types** — structured, queryable records (a POI, an Event) that components render.

Rule: **structured/queryable things** are data content types; **presentation** is Visual Builder
components that query/reference that data. Every routable type carries `SeoMetadata` (see SEO.md).

## Content types (Visit Dubai)

### Routable pages
| Type | URL | Purpose |
|------|-----|---------|
| `HomePage` | `/` | Landing (Visual Builder experience) |
| `PlacesToVisitListingPage` | `/places-to-visit` | Points-of-interest index |
| `EventListingPage` | `/events` (or `/whats-on`) | What's on index |
| `ArticleListingPage` | `/articles` (or `/guides`) | Editorial guides/news index |
| `TourListingPage` | `/tours` | Tours & experiences index |
| `HotelListingPage` | `/hotels` (or `/stay`) | Places to stay index |
| `AreaPage` | `/areas/{slug}` | Neighborhood/area landing |
| `SearchPage` | `/search` | Semantic search UI |

### Structured content
| Type | Purpose | Key fields |
|------|---------|-----------|
| `PointOfInterest` (Places to Visit) | A place to visit / thing to do (POI) | name, slug, summary, body, images, `area` (ref), `categories` (tags), geo (lat/lng), priceBand, openingHours, SEO → `TouristAttraction` |
| `Event` | A dated event | name, slug, summary, body, images, startDate, endDate, venue/`area` ref, `categories`, ticketUrl, SEO → `Event` |
| `Article` | Editorial guide / news | title, slug, excerpt, body (rich text), heroImage, author, publishDate, `categories`, related refs, SEO → `Article`/`NewsArticle` |
| `Tour` | A curated experience / package | name, slug, summary, body, images, duration, priceFrom, highlights, stops (`PointOfInterest[]`), `categories`, bookingUrl, SEO → `TouristTrip` |
| `Hotel` | A place to stay | name, slug, summary, body, images, `area` ref, starRating, priceBand, amenities, geo, bookingUrl, SEO → `Hotel`/`LodgingBusiness` |
| `Area` | Neighborhood/region | name, slug, description, heroImage, geo, SEO → `Place` |
| `Itinerary` | Multi-day plan (also the AI Trip Planner output) | title, slug, days[{ dayNo, items[ref to POI/Event/Tour/Hotel], notes }], summary, heroImage, SEO |

### Taxonomy / shared
| Type | Purpose |
|------|---------|
| `Category` (tag) | Cross-cutting (beach, culture, dining, adventure, family, luxury, desert…) |
| `SeoMetadata` (component) | title, description, canonical, ogImage, noindex — see SEO.md |
| `MediaAsset` | image/video + alt text + credit/attribution (logged in ASSETS.md) |

### Visual Builder components (elements)
Hero · SectionHeading · POICardGrid · EventCarousel · ArticleGrid · TourGrid · HotelGrid ·
FeaturedCollection · RichText · ImageGallery · MapEmbed · CTABanner · Breadcrumbs · SearchBar ·
StatStrip · Newsletter/CTA · Accordion/FAQ · ItineraryTimeline.

Each maps 1:1 to a React component via the **official `@optimizely/cms-sdk`** content-type→
component mapping (generated by the `optimizely-model-react` skill), and follows
**COMPONENT-STANDARDS.md** — including which components expose **Light/Dark** and
**Full-Width/Container** display settings (theme/width live on Sections; a few banner-type
components own them). The 12-column grid + `<SectionShell>`/`<Container>`/`<Grid>` primitives are
the layout contract.

## Relationships
- `PointOfInterest` → `Area` (many-to-one), → `Category[]`
- `Event` → `Area`/venue, → `Category[]`
- `Tour` → `PointOfInterest[]` (stops), → `Category[]`
- `Hotel` → `Area`, → `Category[]`
- `Article` → `Category[]`, → related `PointOfInterest[]`/`Event[]`/`Tour[]`
- `Itinerary` → mixed refs (POI/Event/Tour/Hotel) per day
- Listing pages query their type via Graph with filter + facet on `Category`, `area`, dates, price.

## Graph query surface
- **Listing:** filtered + paginated + faceted queries per type (facets: category, area, price
  band, date).
- **Detail:** fetch by `slug` / `_metadata.url` (use singular `item` for caching).
- **Semantic search:** `_fulltext { match }` + `orderBy: { _ranking: SEMANTIC, _semanticWeight:
  0.3–0.5 }` across POI/Event/Article/Tour/Hotel; return a **unified, type-tagged card model**.
- **Localization:** `locale` arg (EN now; AR later — see BRAINSTORM/Opal note).
- **Multisite (if used):** scope via a content field in `where` (e.g. `SiteId`).
